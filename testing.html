Test Plan

This project involves several pieces that require testing: the card identification model, the back-end server, front-end application, whether they are cross-platform capable, various hardware and software combinations, and whether the interface is user friendly and bug free. This is the team's plan to resolve most, if not all, issues related to that.

For those unfamiliar with Euchre, here is a link describing the game and how to play.

Testing

  • Unit Testing
    1. Model
      1. Test against an image with no cards in it. Does it return “None”?
      2. Test every individual card, i.e. 9-A of all four suits. Does it identify them correctly?
      3. Test an image of a Euchre Hand. Does it identify every truncated image correctly?

        For example, can it properly identify each card in the image below. Notice that only parts of the cards are visible here.

      4. Test an image with multiple Euchre hands. Does it identify every card image correctly?

        For example, can it identify the cards in the image below?

      5. Test against various Euchre hands and combinations of known card types. Does it identify every card correctly?
      6. Test against cards that are not included in a Euchre deck, like 8 of Hearts. Does it return “None”?
      7. Test against an image with every Euchre card in it. Does it recognize all of the cards?
      8. Test against non-standard card images. Does it still recognize the card?

        For example, can it identify the image given below.

    2. Server
      1. Feed an image to the server in. Does it respond? Does it send back an image?
      2. Send a non-image file to the server. Does it respond? Does it handle invalid input by sending an error message?
      3. Send an image to the server, wait for its response, send corrections to the server. Does it make the corrections and send them back to the user?
      4. Send a NULL input to the server. Does it handle it appropriately/gracefully?
      5. Test how long it takes the server to respond to a request. Should be within 5-10 seconds for smaller image resolutions.
    3. Web Application
      1. Attempt to send an empty input to the server. Does it disallow an empty input? Does it require a file before sending to the server?
      2. Attempt to send a file without specifying the trump. Does it not allow the user to do so?
      3. Attempt to send an invalid file type, like .xml file to the server. Does it not allow the user to do so?
      4. Is the web page viewable to users on the www, i.e. can one access it from outside the local machine?
      5. Is it viewable/readable on a mobile device?
  • Integration/Functional Testing
    1. Go to web page. Is it available to users, i.e. can one access it outside of the local machine?
    2. Send image and trump suit to server. Does it respond? Does it reply with the card labels and the labelled cards in the image?
  • System Testing
    1. Windows 10
      1. Google Chrome
      2. Microsoft Edge
      3. Mozilla Firefox
    2. Mac OSX
      1. Safari
      2. Google Chrome
      3. Mozilla Firefox
    3. Mobile
      1. Safari
      2. Google Chrome (ios)
      3. Default Android Browser
      4. Google Chrome (Android)
  • User/Acceptance Testing

    Since different group members will have more experience developing certain parts of the application than others, we will have those less experienced with the web application for testing that end of it and providing feedback to the most experienced members of each component. On top of that, we will have a few peers outside the team testing the application to see if they can find any issues.

    The model will be tested through automated evaluation jobs that can be run on the cloud in order to speed up the evaluation and collect significant data. This will not have a direct user component (except for preparing the images to test with).
